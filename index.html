Payments API Reference
— POST /v1/payments/initiate

Sample API reference documentation for a fictional fintech payments platform (PayAxis). Demonstrates endpoint documentation, parameter tables, request/response examples, error handling, and code samples — structured to OpenAPI standards.

Overview

The POST /v1/payments/initiate endpoint creates a new payment transaction. On success, the endpoint returns a payment_id and initial status. Use the payment_id to poll GET /v1/payments/{payment_id} for status updates, or configure a webhook to receive real-time callbacks.

Authentication All requests to /v1/payments/* require a Bearer token in the Authorization header. Tokens are scoped at the API key level. Use a secret key (prefix: sk_) for server-side requests only. Never expose secret keys in client-side code or version control.

Base URL

HTTP
https://api.payaxis.io/v1

Request

Headers

Idempotency-KeyUUID v4 string. Prevents duplicate charges on retried requests.Optional

Body Parameters

ParameterTypeRequiredDescription
amountintegerRequiredPayment amount in the smallest currency unit (e.g., kobo for NGN, cents for USD). Minimum: 100.
currencystringRequiredISO 4217 currency code. Supported: NGN, USD, GBP, EUR.
customer_idstringRequiredPayAxis customer identifier (prefix: cust_). Must be a verified customer object.
payment_methodstringRequiredPayment method type. Accepted values: card, bank_transfer, ussd, wallet.
descriptionstringOptionalHuman-readable description of the transaction. Max 255 characters. Appears on customer receipts.
metadataobjectOptionalArbitrary key-value pairs for internal reference. Not returned to customers. Max 10 keys, 500 characters per value.
callback_urlstringOptionalHTTPS URL to receive webhook notifications on status change. Overrides the default webhook URL configured in your dashboard.

Idempotency Always supply an Idempotency-Key for payment initiation requests. Network timeouts may cause your client to retry a request that already succeeded server-side. Without an idempotency key, the retry will create a duplicate charge.

Getting Started with FlowDesk
— Send Your First Automated Email in 10 Minutes

Sample quickstart documentation for a fictional SaaS email automation platform (FlowDesk). Demonstrates task-oriented structure, prerequisite clarity, numbered steps, and outcome-focused writing.

Prerequisites

Before you begin, ensure you have the following:

  • A FlowDesk account (free trial or paid). Sign up here if you haven't already.
  • A verified sender domain. Domain verification takes 5–10 minutes. See Domain Verification Guide.
  • At least one contact in your FlowDesk audience, or a CSV file ready to import.
Estimated Time This guide takes approximately 10 minutes to complete. By the end, you will have created an audience, built a simple automation flow, and sent a test email.

Step 1 — Create Your Audience

An Audience is the contact list your automation will send to. FlowDesk segments contacts by audience, so you need at least one before creating a flow.

  1. In the left sidebar, click Audiences, then click Create Audience.
  2. Enter an audience name (e.g., New Signups) and an optional description.
  3. Under Default Fields, select the contact properties to collect: at minimum, select email and first_name.
  4. Click Save Audience. You'll be redirected to the audience detail view.
  5. To import contacts, click Import Contacts and upload a CSV with columns matching your selected fields. Click Confirm Import.
Checkpoint Your audience is ready when you see at least one contact listed in the Contacts tab. You can proceed with zero contacts if you plan to use the Send Test feature in Step 3.

Step 2 — Build Your First Flow

A Flow is a sequence of automated actions triggered by an event. In this guide, you'll create a welcome email triggered when a contact is added to your audience.

  1. In the left sidebar, click Flows, then click New Flow.
  2. Select Start from scratch. The Flow Builder canvas opens.
  3. Click the + icon on the canvas to add a trigger. Select Contact Added to Audience and choose the audience you created in Step 1.
  4. Click the + icon below the trigger to add an action. Select Send Email.
  5. In the email editor panel on the right, fill in:
    Subject: Welcome to [Your Brand] — you're in. 🎉
    From Name: Your brand name
    From Email: A verified sender address
  6. Click Edit Email Body to open the drag-and-drop editor. Add your welcome message. Use the {{first_name}} merge tag to personalise the greeting.
  7. Click Save & Close to return to the canvas.
  8. Click Activate Flow in the top-right corner.
Important A flow only processes new trigger events after activation. Contacts already in the audience before activation will not receive the welcome email unless you use the Backfill option in Flow Settings.

Step 3 — Send a Test Email

Before going live, always send a test to verify personalisation, formatting, and links.

  1. In the Flow Builder, click the Send Email action block to select it.
  2. In the right panel, click Send Test Email.
  3. Enter your email address in the Test Recipient field. Optionally, set a first_name value to preview personalisation.
  4. Click Send Test. Check your inbox — delivery typically takes under 60 seconds.

Verify Delivery

To confirm the test arrived and rendered correctly, check:

  • Subject line displays correctly in your inbox preview
  • Personalisation tags ({{first_name}}) resolved as expected
  • All links are clickable and point to the correct URLs
  • Email renders correctly on both desktop and mobile (use the Preview Modes button in the editor)
You're Live Once your flow is activated and your test looks correct, your welcome email will automatically send to every new contact added to the audience — with no further action required.

Next Steps

  • Add a delay step between emails to build a multi-email onboarding sequence
  • Use Conditions to branch your flow based on contact properties or email engagement
  • Connect FlowDesk to your product via the Events API to trigger flows on in-app actions

Release Notes
— FlowDesk v3.4.0 · June 2025

Sample release notes for FlowDesk v3.4.0. Demonstrates the standard changelog format: version header, summary, categorised changes (new features, improvements, bug fixes, deprecations), migration notes, and known issues.

Release Summary v3.4.0 introduces native Webhooks v2 with retry logic, a redesigned Flow Analytics dashboard, and performance improvements to the Contact Import pipeline. This release also deprecates the legacy /v1/lists endpoint — see Migration Notes below.

Release Details

FieldValue
Version3.4.0
Release DateJune 1, 2025
TypeMinor release — backwards compatible
Affected PlansAll plans (Starter, Growth, Enterprise)

New Features

Webhooks v2 with Automatic Retry

Outbound webhooks now include automatic retry logic. If a webhook delivery fails (non-2xx response or timeout), FlowDesk retries up to five times using exponential backoff: 1 min, 5 min, 30 min, 2 hr, 6 hr.

  • Delivery logs are available in Settings → Webhooks → Delivery History
  • Failed deliveries after all retries are marked failed and trigger an email alert to the account owner
  • Signature verification headers (X-FlowDesk-Signature) are unchanged
Action RequiredYour existing webhook endpoints do not need to change. Retry logic is applied automatically to all configured endpoints.

Flow Analytics Dashboard — Redesign

The Flow Analytics view has been rebuilt with improved data visualisation and two new metrics:

  • Revenue Attributed: Tracks conversions and estimated revenue driven by each flow, when FlowDesk eCommerce tracking is enabled
  • Step-Level Drop-off: Shows the percentage of contacts that exit the flow at each step, allowing targeted optimisation

Improvements

  • Contact Import Performance: Import processing time for large CSV files (>100,000 rows) reduced by approximately 60% through parallel processing pipeline.
  • Flow Builder: Canvas now supports keyboard navigation. Use arrow keys to move between nodes, Enter to open a node's settings panel, Delete to remove a selected node.
  • API Rate Limits: Rate limit headers (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset) are now returned on all API responses, not only on 429 errors.
  • GDPR Data Export: Contact data export requests now complete within 2 hours (previously up to 24 hours) and include a full history of email interactions.

Bug Fixes

  • Fixed: Merge tags containing special characters (&, <, >) were being HTML-encoded in plain-text email parts. Now rendered correctly. [#FD-4821]
  • Fixed: Flow activation toggle was unresponsive for users with editor role on Enterprise plans. [#FD-4903]
  • Fixed: Duplicate contacts could be created when the same email address was submitted via the API and the embed form simultaneously within a 500ms window. [#FD-4754]
  • Fixed: GET /v2/flows/{flow_id}/analytics returned a 500 error for flows with zero send events. Now returns an empty analytics object with zero values. [#FD-4867]

Deprecations

Deprecation Notice — /v1/lists The /v1/lists endpoint family is deprecated as of v3.4.0 and will be removed in v4.0.0 (planned: September 2025). Migrate to /v2/audiences. The v2 Audiences API is functionally equivalent with expanded filtering and pagination support. See the Migration Guide for a complete endpoint mapping.

Known Issues

  • Flow Analytics — Revenue Attributed: Revenue attribution is not yet available for multi-currency accounts. Affected accounts will see a "Coming Soon" placeholder. Expected in v3.5.0. [#FD-4991]
  • Safari 16: The Flow Builder canvas may display incorrectly on Safari 16 on macOS Ventura. Workaround: Use Chrome or Firefox. A fix is in progress. [#FD-5012]

FlowDesk Security Overview
— Data Protection, Access Controls & Compliance

Sample security documentation for enterprise and compliance audiences. Demonstrates infrastructure security, data handling, access control, and regulatory alignment — written to enterprise security review standards.

This document summarises FlowDesk's security architecture, data protection practices, and compliance posture. It is intended for security teams, IT procurement reviewers, and compliance officers evaluating FlowDesk for enterprise deployment.

Last Reviewed This document was last reviewed and updated: June 2025. For the most current security certifications and penetration test summaries, contact security@flowdesk.io.

Infrastructure & Hosting

FlowDesk's production infrastructure is hosted on Amazon Web Services (AWS), with primary workloads in the eu-west-1 (Ireland) and us-east-1 (Virginia) regions. Customer data residency is configurable at the account level for Enterprise plans — contact your account manager to confirm your data region.

ControlImplementation
Data Encryption at RestAES-256 encryption via AWS KMS. All customer data, including contact records, email content, and flow configurations, is encrypted at rest.
Data Encryption in TransitTLS 1.2 minimum enforced on all external connections. TLS 1.3 supported and preferred. HTTP connections are rejected or redirected to HTTPS.
Database SecurityRDS instances are deployed in private VPC subnets with no public internet access. Access requires VPN or bastion host authentication.
Secrets ManagementAll credentials, API keys, and environment secrets are managed via AWS Secrets Manager and rotated automatically on a 90-day cycle.

Access Controls

FlowDesk implements role-based access control (RBAC) at both the application and infrastructure level.

Application-Level RBAC

RolePermissions
OwnerFull access to all settings, billing, user management, and data export. Maximum one Owner per account.
AdminFull access except billing and account deletion. Can manage team members up to their own permission level.
EditorCan create, edit, and activate flows, audiences, and templates. Cannot access billing, API keys, or user management.
ViewerRead-only access to flows, analytics, and audience data. Cannot create or modify any objects.

Infrastructure-Level Access

  • Production environment access is restricted to authorised engineering staff via MFA-enforced SSO
  • All production access sessions are logged to an immutable audit trail in AWS CloudTrail
  • Direct database access requires a time-limited credential issued through a privileged access management (PAM) system
  • The principle of least privilege is enforced — no role has broader access than required for its function

Responsible Disclosure FlowDesk operates a responsible disclosure programme. If you discover a potential security vulnerability, report it to security@flowdesk.io. Do not disclose vulnerabilities publicly before FlowDesk has had a reasonable opportunity to investigate and remediate.

Incident Response

FlowDesk maintains a documented Incident Response Plan (IRP) aligned to NIST SP 800-61. In the event of a confirmed data breach affecting customer data, affected customers will be notified within 72 hours in accordance with GDPR Article 33 requirements. Notification will include the nature of the breach, data categories affected, likely consequences, and remediation actions taken.

SOP: Handling Customer-Reported
Email Deliverability Issues

Sample internal SOP for a SaaS customer support team. Demonstrates process documentation, decision logic, clear ownership, and actionable steps — structured for rapid onboarding and consistent execution.

FieldValue
SOP IDCS-OPS-014
Version2.1
OwnerCustomer Support Lead
Last UpdatedMay 2025
Applies ToAll Customer Support Representatives (Tier 1 and Tier 2)

Purpose

This SOP defines the standard process for diagnosing and resolving customer-reported email deliverability issues, including emails landing in spam, low open rates due to deliverability failures, and bounced sends. Following this process ensures consistent first-response quality, reduces escalation time, and maintains customer trust during deliverability incidents.

Scope

This SOP covers deliverability issues reported through the support ticket system. It does not cover:

  • Bulk spam complaints affecting platform-wide sending reputation (handled by the Deliverability Engineering team)
  • Account-level sending suspensions (handled under SOP CS-OPS-021)

Procedure

Phase 1 — Initial Triage (Target: within 30 minutes of ticket creation)

  1. Open the customer's account in the admin panel (Admin → Accounts → [Account ID]). Confirm account is active and not suspended.
  2. Navigate to Deliverability → Send Logs. Filter by the date range and audience segment the customer reported. Identify the affected sends by status: delivered, soft_bounce, hard_bounce, spam_complaint, or deferred.
  3. Check Domain Verification Status under Settings → Sending Domains. A domain showing SPF: fail, DKIM: fail, or DMARC: none is almost certainly the cause. Proceed to Phase 2 (Domain Authentication).
  4. If domain authentication is passing, check the Bounce Rate for the affected audience. A bounce rate above 5% on a single send triggers the High Bounce Rate sub-procedure (see Section 4.2).
  5. Send the customer an initial response within 30 minutes acknowledging the report, confirming you are investigating, and setting a resolution ETA of 4 business hours.

Phase 2 — Domain Authentication Issues

  1. Send the customer the Domain Authentication Troubleshooting Guide (internal link: docs/deliverability/domain-auth). Include their specific failing record in the ticket response.
  2. If the customer is on a Growth or Enterprise plan, offer a live screen-share session to walk through the DNS configuration in real time. Book via Calendly (link in the support team Notion).
  3. After the customer confirms DNS changes have been made, use the built-in DNS checker (Admin → Tools → DNS Checker) to verify propagation. DNS propagation can take up to 48 hours — set a follow-up reminder in the ticket system.
  4. Once all records show pass, ask the customer to send a test email and confirm delivery to a Gmail and an Outlook address (these represent the majority of inbox providers where deliverability failures are reported).
Escalation Trigger Escalate to Tier 2 if: (a) domain authentication is passing but deliverability issues persist, (b) the customer reports a sudden drop in open rates exceeding 20 percentage points, or (c) multiple accounts report simultaneous deliverability failures (possible platform-wide IP reputation issue).

Escalation Path

ScenarioEscalate ToSLA
Authentication passing, delivery still failingTier 2 Support2 hours
Suspected IP blacklistingDeliverability Engineering1 hour
Customer threatening churn or legal actionCustomer Support Lead + Account Manager30 minutes

Closure & Documentation

  1. Once the issue is resolved and confirmed by the customer, update the ticket status to Resolved.
  2. Log the root cause in the Resolution Notes field using the standardised taxonomy: domain-auth, high-bounce, spam-content, ip-reputation, platform-bug, or user-error.
  3. If the issue revealed a product gap or documentation gap, file a Jira ticket under the CS-FEEDBACK project with label deliverability and link it to this support ticket.
Quality Check Before closing any deliverability ticket, confirm: (1) the customer has sent a successful test email, (2) the root cause is documented in Resolution Notes, and (3) any follow-up DNS verification reminders have been cancelled or completed. Incomplete closure is the primary cause of deliverability re-opens.
